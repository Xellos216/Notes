# 11_gitignore_reasons.md

## 📌 왜 .gitignore에 IDE/빌드 파일을 추가해야 할까?

이 문서는 `.idea/`, `*.iml`, `out/` 등의 파일을 GitHub에 올리지 않아야 하는 이유를 실무적 관점에서 상세히 설명합니다.

---

## 1. `.idea/` 디렉토리
> IntelliJ IDEA의 사용자별 프로젝트 설정 폴더

| 문제 | 설명 |
|------|------|
| 🧩 개인 설정 혼재 | 사용자마다 다른 설정(경로, 테마 등)이 섞여 협업에 문제 발생 |
| ⚠ 협업 시 충돌 | workspace.xml 등 충돌 위험이 높고 불필요한 변경 감지 |
| 🛠 의미 없는 추적 | Git이 코드 외 설정을 계속 추적하게 됨 |

---

## 2. `*.iml` 파일
> IntelliJ의 모듈 정보 파일

| 문제 | 설명 |
|------|------|
| 🔄 자동 생성 가능 | IntelliJ가 자동으로 생성하며 Git 추적 필요 없음 |
| 💥 이름 충돌 가능 | 디렉토리 이름 변경 시 모듈 충돌 가능 |
| 🧪 Gradle과 중복 | Gradle/Maven에서 모듈 정보가 이미 정의되어 있음 |

---

## 3. `out/` 디렉토리
> Java 컴파일 결과물이 저장되는 빌드 폴더

| 문제 | 설명 |
|------|------|
| 🎯 컴파일 결과물은 Git 추적 대상 아님 | 바이너리는 배포 결과일 뿐 소스가 아님 |
| 💾 용량 낭비 | `.class` 등 바이너리가 GitHub에 올라가 저장 공간 낭비 |
| 🚫 환경 의존성 | JDK 버전이나 OS에 따라 결과물이 달라져 일관성 깨짐 |

---

## ✅ 결론: .gitignore로 소스코드와 환경 설정을 분리하자

```gitignore
# IDE 설정
.idea/
*.iml

# 빌드 결과물
out/
```

- 협업 충돌 방지
- 코드 외 파일 추적 방지
- GitHub를 깔끔하게 유지

이 원칙은 모든 언어와 IDE에 공통으로 적용되는 베스트 프랙티스입니다.

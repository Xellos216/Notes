# SQL Day 1 - Q&A 정리

## Q1 SQL은 정보 보관하는 통인가요?
**A1** SQL은 언어이고, 정보를 보관하는 통은 데이터베이스(DB)입니다. SQL은 그 DB를 다루기 위한 도구입니다.

## Q2 로컬 DB를 다른 곳에 넘겨줄 수 있나요?
**A2** 가능합니다. `mysqldump` 같은 명령어로 내보내서 다른 서버에서 복원할 수 있습니다.

## Q3 외부 공격으로 SQL이 털리면 DB 전체가 털리나요?
**A3** 그럴 수 있습니다. SQL Injection 같은 공격은 잘못된 입력으로 DB 쿼리를 조작하여 민감한 정보에 접근하게 만듭니다.

## Q4 Query는 무엇인가요?
**A4** Query는 DB에게 "요청하는 명령"을 의미합니다. SELECT, INSERT, UPDATE 같은 SQL 문장이 모두 Query입니다.

## Q5 SQL 명령어는 왜 대문자가 많은가요?
**A5** 가독성과 관례 때문입니다. SQL은 대소문자 구분을 안 하지만, 문법은 대문자, 테이블/컬럼은 소문자로 구분하는 스타일이 많습니다.

## Q6 데이터베이스와 테이블의 차이?
**A5** 데이터베이스는 여러 테이블을 담는 상위 개념입니다. 비유하면 데이터베이스는 폴더, 테이블은 파일, 각 행은 엑셀 줄이라 보면 됩니다.

## Q7 지금 내가 수동으로 테이블 만들고, 제약조건 걸고, 데이터를 넣고 있는데... 이게 진짜 실무에서 어떤 역할을 해?

**A7** 아주 좋은 질문. 네가 이해한 흐름은 거의 맞아:

**A7**-1. **MySQL로 규칙(스키마)** 를 정의한다

* 실무에서도 `id는 비어 있으면 안 됨`, `email은 중복되면 안 됨` 같은 규칙이 꼭 필요함.
* 이걸 SQL로 직접 `CREATE TABLE`하거나, Spring/JPA 같은 프레임워크가 자동으로 생성해줌.

---

**A7**-2. **웹사이트와 MySQL은 백엔드 코드로 연결된다**

* 예: Spring 서버 → MySQL
* 사용자가 웹사이트에서 `회원가입` 폼을 작성하면,
  백엔드 서버가 그걸 받아서 SQL INSERT를 실행함.
* 이때도 `NOT NULL`, `UNIQUE`, `FOREIGN KEY` 같은 제약조건이 실제로 작동해서 유효하지 않은 데이터를 막아줌.

---

**A7**-3. **웹사이트와 DB는 파일처럼 붙어 있지 않음**

* 웹 서버(Spring)와 DB는 **연결(connection)** 을 통해 통신함.
* MySQL 서버는 보통 포트 `3306`에서 동작하고, 웹 서버가 여기에 접속함.
* 두 서버는 물리적으로 분리돼 있어도 OK (예: 클라우드 DB 사용)

---

**A7**-4. **지금은 수동 실습이지만, 실무에서는 백엔드가 자동으로 처리한다**

* 사용자가 `폼`을 작성 → 서버에서 `SQL INSERT`
* 개발자는 제약조건이 잘 작동하는지 테스트하기 위해 지금처럼 수동 실습을 해보기도 함.
* 예: 테이블 설계 이후에는 `제약조건 테스트용 SQL 스크립트`를 따로 짜서 배포 전에 검증

---

### ✅ 정리하면:

| 네 이해            | 실제 실무와 비교                       | 일치도               |
| --------------- | ------------------------------- | ----------------- |
| 규칙 정의 → 연결      | 테이블 구조 정의 후 백엔드 연결              | ✅                 |
| 사용자 입력 저장       | 폼 입력 → 서버 코드 → DB 저장            | ✅                 |
| 웹과 DB가 한 폴더에 있음 | 보통 분리된 서버, 연결로 통신               | ❌ (거의 맞음, 개념은 OK) |
| 수동 SQL 입력은 연습용  | 실무에선 자동 처리되지만, 로직 테스트 시 수동도 활용함 | ✅                 |

## Q8 SELECT 문에서 u, o, p는 무엇을 의미하나요?

**A8**

`u`, `o`, `p`는 각각 테이블의 **별칭(alias)** 입니다:

- `o` → `orders_v2` 테이블
- `u` → `users` 테이블
- `p` → `products` 테이블

이 별칭을 통해 쿼리에서 각 테이블을 간결하게 참조할 수 있습니다.

---

## Q9 INNER / LEFT / RIGHT JOIN은 왜 그렇게 동작하나요?

**A9** 각 JOIN은 **"누가 기준이냐"**에 따라 포함되는 행이 달라집니다. 단순히 외우는 게 아니라 구조적으로 정해진 동작 방식입니다.

---

**A9**-1: INNER JOIN: 교집합만 출력

- **조건을 만족하는 데이터(양쪽 테이블에 모두 존재하는 값)** 만 결과에 포함
- 예: 주문이 있는 회원만 보고 싶을 때

---

**A9**-2: LEFT JOIN: 왼쪽 기준 전체 + 오른쪽 매칭

```sql
FROM users u
LEFT JOIN orders_v2 o ON ...
```

- 모든 회원은 결과에 포함
- 주문이 없는 회원은 `NULL`로 표시됨
- 예: "주문 안 한 회원 포함 전체 목록"을 보고 싶을 때

---

**A9**-3 RIGHT JOIN: 오른쪽 기준 전체 + 왼쪽 매칭

```sql
FROM products p
RIGHT JOIN orders_v2 o ON ...
```

- 모든 주문은 결과에 포함
- 주문된 상품 정보가 없으면 `NULL`
- 예: "전체 주문 기준으로 상품 매칭 여부" 확인

---

### 📌 핵심 요약

| JOIN 종류    | 기준 테이블   | 포함 범위              | 용도 예시                        |
|--------------|----------------|-------------------------|-----------------------------------|
| `INNER JOIN` | 공통 기준       | 양쪽 테이블 모두 일치   | 실제 주문 내역 추출              |
| `LEFT JOIN`  | 왼쪽 테이블     | 왼쪽 전부 + 오른쪽 일부 | 전체 회원 중 미주문자 확인       |
| `RIGHT JOIN` | 오른쪽 테이블   | 오른쪽 전부 + 왼쪽 일부 | 전체 주문 기준 상품/회원 분석    |

## Q10 `JOIN orders_v2 o ON u.id = o.user_id`에서 ON은 뭐야?

**A4**** `ON`은 **조인 조건**을 명시하는 절이다.
`u.id = o.user_id`는 `users` 테이블의 기본키(id)와 `orders_v2`의 외래키(user\_id)를 연결하여 조인하겠다는 의미다.

## Q11 SQL은 왜 읽는 순서(위→아래)와 실행 순서가 다른 거야?

**A11**** SQL은 **선언형(Declarative)** 언어이기 때문이다.

* Kotlin, Python 같은 절차형 언어는 순서대로 명령을 실행하지만,
* SQL은 \*\*“무엇을 얻고 싶은지”\*\*만 선언하면, 내부적으로 DB 엔진이 최적화된 순서로 실행한다.

▶ 예를 들어, SQL은 `SELECT → FROM → WHERE` 순으로 쓰이지만, 실제 실행 순서는 `FROM → JOIN → WHERE → GROUP BY → HAVING → SELECT`이다.

이 구조는 사람이 읽고 이해하기 쉽게 설계된 것이며, DB는 실행 최적화를 위해 순서를 유연하게 해석한다.

## Q12 지금 CLI로 MySQL 실습 중인데 DBeaver 같은 프로그램을 안 쓰고 CLI 환경에서 SQL을 실습했을 때 얻는 이점이 뭐야?

### ✅ A12:
CLI 환경에서 SQL을 실습하면 다음과 같은 장점이 있어:
1. **SQL 문법과 구조에 더 집중할 수 있음**  
   → GUI의 자동완성, 시각화 기능 없이 순수하게 SQL 문장 구성 연습이 가능.

2. **운영 환경과 유사한 경험**  
   → 실제 서버에서는 대부분 CLI 기반(MySQL Shell 또는 터미널 접속)으로 쿼리를 실행하므로 실무 감각과 가까움.

3. **스크립트 자동화/버전 관리에 유리함**  
   → SQL 문장을 `.sql` 파일로 관리하고 git 등으로 버전 관리 가능.

4. **리눅스 기반 도구와의 연계가 쉬움**  
   → Bash, `awk`, `sed`, `less` 등의 도구와 조합하여 처리 가능 (e.g. 백업, 로그 필터링 등)

## Q14 orders_v2 테이블에서 price가 없는데 SUM이 안 되는 이유?

**현상**:  
`SELECT SUM(price) FROM orders_v2;` 실행 시 "Unknown column 'price'" 오류 발생

**원인**:  
orders_v2 테이블에는 `price` 컬럼이 존재하지 않음. 가격 정보는 `products` 테이블에 있음.

**해결**:  
`orders_v2`와 `products` 테이블을 `JOIN`한 후 `products.price * quantity`로 계산해야 함.

## Q16 WHERE 절과 HAVING 절의 차이는 정확히 무엇인가?

**A16**

|----------|-------------------------|-------------------------------|
| 적용 시점 | GROUP BY 이전           | GROUP BY 이후                 |

| 예시     | quantity > 1            | COUNT(*) > 1                 |

## Q17 JOIN 뒤에 INNER, LEFT를 명시하지 않아도 되는 이유?

**A17**
**설명**:  
`JOIN`은 SQL에서 `INNER JOIN`과 동일하게 작동함. 즉, 양쪽 테이블에 매칭되는 데이터만 추출됨.

**예시**:
```sql
-- 동일한 결과
SELECT * FROM A JOIN B ON 조건;
SELECT * FROM A INNER JOIN B ON 조건;
```

**추가 참고**:
- `LEFT JOIN`: 왼쪽 테이블 기준으로 모두 출력
- `RIGHT JOIN`: 오른쪽 테이블 기준으로 모두 출력
- `FULL OUTER JOIN`: 양쪽 전체 포함 (MySQL에서는 직접 지원 안 함)

## Q18 서브쿼리 사용 시 JOIN 순서가 바뀌는 이유?

**A18**
**현상**:  
서브쿼리를 먼저 작성하고, 그 결과에 대해 JOIN하거나 정렬해야 하는 구조가 낯설게 느껴짐.

**이유**:  
서브쿼리는 먼저 실행되어 **임시 테이블처럼 작동**함.  
따라서 JOIN, ORDER BY는 바깥 쿼리에서 처리하는 것이 논리적으로 맞음.

**정리**:
- 안쪽 서브쿼리: 집계 수행 (`GROUP BY`, `SUM`, `COUNT` 등)
- 바깥쪽 쿼리: 필터링, 정렬, JOIN 등 추가 조작

## Q19 인덱스를 만들었는데 `EXPLAIN` 결과가 그대로인 이유?

**A19**
**현상**:  
`EXPLAIN SELECT * FROM users WHERE email = 'alice@exaple.com';` 실행 결과가 인덱스를 만들기 전과 동일함

**원인**:  
`alice@exaple.com` 오타로 인해 조건에 매칭되는 데이터가 없어 MySQL 옵티마이저가 인덱스를 사용하지 않음

**해결**:  
실제 존재하는 값(`hong@example.com`)으로 조건을 수정하면 `EXPLAIN` 결과에서 `type = const`, `key = unique_email`이 출력되며 인덱스 사용이 확인됨

## Q20 인덱스(Index)의 정확한 의미는?

## Q21
**정의**:  
WHERE, JOIN 조건 등에 자주 쓰이는 컬럼을 빠르게 탐색하기 위한 B-Tree 기반 데이터 구조

**비유**:  
책의 목차처럼, 특정 키워드로 빠르게 jump해서 행(row)의 위치를 찾아주는 역할

---

## Q21 인덱스는 WHERE 조건을 빠르게 찾기 위한 구조물인가?

**A21**
**정답**:  
맞음. 인덱스가 없으면 테이블 전체를 검색해야 하며, 인덱스는 WHERE 조건으로 원하는 row 위치를 빠르게 탐색하게 해줌

## Q24 FOREIGN KEY라는 게 정확히 무슨 의미야?

**A24**
**정의**:  
다른 테이블의 특정 컬럼 값을 참조하도록 강제하는 제약조건.

**예시**:  
`orders_v2.product_id REFERENCES products.product_id`  
→ products에 존재하지 않는 product_id는 입력 불가

**목적**:  
데이터 무결성 보장 (예: 주문이 유효한 상품을 참조해야 함)

---

## Q25 트랜잭션이라는 건 결국 특정 행의 특정 컬럼 값을 내가 수정하는 건가?

**A25**
**답변**:  
트랜잭션은 단순한 수정이 아니라 “작업의 묶음”  
→ 여러 SQL 명령(INSERT, UPDATE, DELETE)을 묶어서 모두 성공 또는 모두 실패 처리

**예시**:  
- 주문 입력 → 포인트 차감 → 재고 차감 → COMMIT
- 중간에 실패하면 전체 ROLLBACK 가능

---

## Q26 FOREIGN KEY와 JOIN은 무슨 차이야?

**A26**
| 항목 | FOREIGN KEY | JOIN |
|------|-------------|------|
| 정의 | 참조 제약조건 | 조회 시 테이블 연결 |
| 동작 시점 | INSERT/UPDATE/DELETE | SELECT |
| 역할 | 무결성 보장 | 데이터 통합 조회 |
| SQL 키워드 | `FOREIGN KEY`, `REFERENCES` | `JOIN`, `ON`, `USING` |

**비유**:  
- FOREIGN KEY = "데이터 규칙자"  
- JOIN = "데이터 조합자"

## Q27 EXPLAIN에서 type, key, Extra 같은 항목은 정확히 뭘 의미해?

**A27**
**답변 요약**:

| 항목 | 의미 |
|------|------|
| type | 접근 방식 (eq_ref > ref > range > index > ALL 순으로 성능 좋음) |
| key | 실제 사용된 인덱스 |
| rows | 예측 스캔 행 수 |
| Extra | 기타 힌트 (Using index, Using where, Using join buffer 등) |

## Q28 트랜잭션 안에서 SELECT하면 반영된 데이터가 보이는 이유?

**A28**
**설명**:
같은 세션에서는 `COMMIT` 전 데이터도 읽을 수 있음.  
하지만 다른 세션에서는 보이지 않음 (격리성 Isolation 성질)

## Q30 트랜잭션 중 실패하면 앞에 성공한 것도 모두 취소되는 게 맞아?

**A30**
